Contexte & Problématique

概要

Dans le cadre d'un projet d'infrastructure réseau d'entreprise, un serveur web Linux a été déployé sur une machine virtuelle Debian 12, placée dans une zone démilitarisée (DMZ). Cette DMZ est isolée du réseau local LAN grâce à un pare-feu pfSense CE 2.7 qui centralise tout le filtrage réseau. Les employés s'authentifient via un portail captif avant d'accéder à l'intranet.

// PROBLÉMATIQUE
Comment déployer et sécuriser un serveur intranet LAMP sous Linux
en l'isolant dans une DMZ via pfSense, tout en contrôlant les accès
des utilisateurs par un portail captif et en garantissant la disponibilité
et la sauvegarde des données ?_

4.1 Installation de la pile LAMP sur Debian 12 ▶

Avant de placer le serveur en DMZ, tous les paquets sont installés pendant que la VM est connectée en NAT (accès Internet). Une fois validés en local, on bascule la carte réseau vers la DMZ.

0 mis à jour, 0 nouvellement installés, 0 à enlever.

tom@lamp:~$ sudo apt install phpmyadmin -y
phpmyadmin est déjà la version la plus récente (4:5.2.1+dfsg-1+deb12u1)

Executing: /lib/systemd/systemd-sysv-install enable mariadb
4.2 Création du portail intranet PHP ▶

Une page PHP dynamique affiche en temps réel : IP du serveur, OS, espace disque, RAM et uptime. Le design adopte un thème terminal (fond noir, texte vert, animation Matrix, journal système animé). Trois versions successives ont été développées.

4.3 Configuration de pfSense ▶

pfSense centralise toute la sécurité réseau : routage, filtrage, DHCP, DNS et portail captif.

// Interfaces réseau assignées
em0 → WAN VMnet8 (NAT VMware) — IP auto — 192.168.8.160
em1 → LAN IP fixe 192.168.10.2/24 — DHCP plage .100–.200
em2 → DMZ IP fixe 192.168.20.2/24 — DHCP désactivé
ALLOW LAN → DMZ (192.168.20.10) TCP 80/443
ALLOW DMZ → WAN Tout (mises à jour)
BLOCK DMZ → LAN Tout — ⚠ Règle critique

⚠ Si le serveur LAMP est compromis, l'attaquant reste bloqué en DMZ — ne peut pas rebondir vers le LAN.

4.4 Mise en réseau DMZ du serveur LAMP ▶

Une fois la pile LAMP validée en local, on bascule la carte réseau VMware vers VMnet2 (DMZ) et on configure l'IP fixe sur Debian.

# /etc/network/interfaces
auto ens33
iface ens33 inet static
address 192.168.20.10
netmask 255.255.255.0
gateway 192.168.20.2
dns-nameservers 192.168.20.2

tom@lamp:~$ sudo systemctl restart networking
tom@lamp:~$ ping 192.168.20.2 -c 4
4 packets transmitted, 4 received, 0% packet loss ✓
tom@lamp:~$ ping 8.8.8.8 -c 4
4 packets transmitted, 4 received, 0% packet loss ✓
4.5 Portail captif pfSense ▶

Le portail captif intercepte toutes les connexions HTTP des clients LAN et les redirige vers une page de login. Seuls les utilisateurs authentifiés peuvent accéder au réseau.

// Services → Captive Portal → LAN_Intranet
Enable: ✓ Enable Captive Portal
Interfaces: LAN
Authentication Method: Use an Authentication backend
Authentication Server: Local Database
Local Auth Privileges: ✓ Allow only users with "Captive Portal Login"

// System → User Manager
Utilisateur: employe1
Privilège: Captive Portal Login ✓
4.6 Résolution DNS locale (intranet.local) ▶

Le DNS Resolver de pfSense est configuré pour résoudre intranet.local → 192.168.20.10. Tous les clients LAN bénéficient automatiquement de cette résolution via le DHCP pfSense, sans modification du fichier hosts.

// Services → DNS Resolver → Host Override
Host: intranet
Domain: local
IP Address: 192.168.20.10
Description: Serveur LAMP DMZ

C:\Users\tom> nslookup intranet.local
Serveur : pfSense.home.arpa
Address: 192.168.10.2
Nom : intranet.local
Address: 192.168.20.10 ✓
4.7 Sauvegarde automatique des bases de données ▶

Un script bash utilise mysqldump pour exporter toutes les bases dans un fichier SQL horodaté. La tâche cron planifie l'exécution chaque nuit à minuit.

# ~/backup_db.sh
#!/bin/bash
DATE=$(date +%Y-%m-%d_%Hh%M)
mysqldump -u root -p --all-databases > ~/backups/backup_$DATE.sql
echo "Sauvegarde terminée le $DATE dans ~/backups/"

tom@lamp:~$ crontab -e
0 0 * * * /bin/bash ~/backup_db.sh # tous les jours à minuit

tom@lamp:~$ crontab -l
0 0 * * * /bin/bash ~/backup_db.sh ✓

Difficultés & Solutions

課題
! Portail captif — Access Denied avec employe1 ▶

Cause : Le privilège "Captive Portal Login" n'était pas attribué à l'utilisateur dans pfSense.

Solution : System → User Manager → Edit employe1
→ Section Privileges
→ Ajouter : "Captive Portal Login"
✓ Authentification fonctionnelle
! DNS intranet.local non résolu depuis Windows ▶

Cause : Le DHCP VMware sur VMnet1 distribuait un autre DNS que pfSense — conflit de serveurs DHCP.

Solution : Désactivation du DHCP VMware sur VMnet1
→ pfSense seul distribue 192.168.10.2 comme DNS
✓ nslookup intranet.local → 192.168.20.10

Axes d'Amélioration

改善
01

HTTPS / Certificat SSL

Activer a2enmod ssl, créer un certificat auto-signé avec OpenSSL, configurer un VirtualHost HTTPS port 443. En production : Let's Encrypt pour un certificat reconnu gratuitement.

02

Fail2Ban — Protection brute-force

Bloquer une IP pendant 30 minutes après 5 tentatives SSH échouées en 10 minutes. Les bannissements sont injectés dans iptables, synchronisables avec pfSense.

03

VPN OpenVPN intégré à pfSense

Permettre l'accès à l'intranet en télétravail. Chaque employé installe un client VPN — tout le trafic transite chiffré vers pfSense, traité comme s'il était sur le LAN.

04

Supervision Zabbix / Grafana

Surveillance en temps réel : disponibilité Apache/MariaDB, CPU/RAM, espace disque backups, temps de réponse du portail. Alertes email/SMS automatiques en cas d'anomalie.

05

Active Directory / LDAP centralisé

Centraliser tous les comptes utilisateurs. pfSense et tous les services s'authentifient via LDAP. Désactiver un compte dans l'annuaire coupe l'accès partout simultanément.

06

Haute disponibilité — CARP pfSense

pfSense est actuellement un SPOF. Le protocole CARP permet deux instances actif/passif — si le principal tombe, le secondaire prend le relais en quelques secondes, de façon transparente.
